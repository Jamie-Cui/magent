#+date: 2026-03-19

* Status

One committed fix landed earlier; a second larger fix is staged in
=magent-fsm-backend-gptel.el= (uncommitted diff, see below).

* Problem Summary

qwen3-max has a streaming quirk where a single tool call is split into
two entries in gptel's =:tool-use= list:

#+begin_example
chunk 1: name="",        id="call_abc",  arguments='{"path":"..."}'
chunk 2: name="read_file", id="call_abc", arguments=""
#+end_example

This creates two problems:

** Problem 1 — invalid =arguments= in the injected assistant message  [FIXED]

gptel's =[DONE]= handler (=gptel-openai.el= line ~213) sets the LAST
tool's =function.arguments= to the last accumulated =:partial_json=,
which is =""= for the empty-args chunk.  The API then rejects the
second request with HTTP 400: =function.arguments must be JSON=.

*** Fix (in =magent--handle-unknown-tools-a=, uncommitted)

After merging the =name=""= entry into the real tool entry (existing
logic), retroactively patch the already-injected assistant message in
=info :data :messages=:

1. Walk backwards to find the last =role="assistant"= message that has
   =tool_calls=.
2. Build an =id→arguments= map from empty-named entries (they carry
   the real JSON).
3. Copy correct JSON args to the real entry whose arguments are ="".
4. Remove the empty-named entry from the =tool_calls= vector.

Logged as:
: INFO fixed 'read_file' arguments in assistant message
: INFO patched assistant message: removed 1 empty tool_call(s)

** Problem 2 — second turn produces =chunks=0= / empty text response  [UNDER INVESTIGATION]

After problem 1 is fixed the API no longer returns 400 and the second
request is accepted.  However the model's second-turn response arrives
with =chunks=0 content-len=0 final-len=0=, so the final response shown
to the user is empty.

Two observed variants:

- *Variant A* (17:05 test): second turn logs =DEBUG reasoning received=
  then =tool-use=nil chunks=0=.  The model produces reasoning but
  empty text — a "pure thinking" response.

- *Variant B* (17:14+, 17:20+ tests): second turn makes *another* tool
  call (=write_file=, =emacs_eval=, =bash= …) with the same split
  quirk.  Our fix handles the split correctly, but the model keeps
  looping.

*** What is confirmed

- =enable_thinking: true= is sent in *every* request because
  =qwen3-max= has =:request-params (:enable_thinking t)= in
  =init-llm.el=.
- The split quirk (empty-name chunk) affects *all* tools, not just
  =read_file=.
- The second request reaches the API (HTTP 200) only after the
  arguments fix.
- When variant A occurs, =state :saw-reasoning= is set but =:text-chunks=
  is empty.  =magent-fsm-backend-gptel--final-text= returns =""=.

*** Hypotheses (ranked)

1. *enable_thinking exhausts output budget* — qwen3-max in extended
   thinking mode uses thinking tokens without a =thinking_budget= cap.
   After a large tool result (e.g. CLAUDE.md, ~2000 lines), the model
   reasons extensively and produces empty visible text.  Testing
   without =enable_thinking= still showed tool loops (variant B), so
   this alone is not the full story.

2. *Conversation structure quirk* — the patched assistant message
   (=content: null=, =tool_calls: [...]= with corrected args) may
   behave differently from a normal first-turn tool call message.
   Perhaps qwen3-max treats =content: null= vs =content: ""= differently
   in multi-turn tool scenarios.

3. *gptel :reasoning-block state leaks between turns* — if
   =:reasoning-block= is ='in= at the end of the first turn (because no
   text content arrived to trigger the ='done= transition), the
   streaming parser for the second turn may silently reclassify text
   content as reasoning via the =(not (eq reasoning-block t))= branch in
   =gptel-request.el= lines 2754–2763.  This would cause text chunks to
   arrive as =(reasoning . text)= callbacks instead of string callbacks,
   leaving =:text-chunks= empty.  *This hypothesis has not been
   conclusively confirmed or ruled out.*

4. *Variant B is actually correct model behaviour being mis-handled* —
   the model makes more tool calls (reasonable) but our =magent-clear-
   session= does not cancel the in-flight gptel FSM, so old sessions
   accumulate and produce confusing interleaved logs.

*** Next steps

1. *Isolate variant A cleanly*: run a single request against a small
   file (e.g. =Makefile=, not =CLAUDE.md=), ensure all previous FSMs are
   dead first (kill curl procs), then capture the *complete* gptel
   debug log for the second-turn response body.  Confirm whether:
   - The HTTP response has =content: ""= (model truly produced no text), or
   - The response has =content: "..."= but gptel drops it due to
     =:reasoning-block= mis-state.

2. *Check =:reasoning-block= state at turn boundary*: add a log line
   just before gptel sends the second request that prints
   =(plist-get info :reasoning-block)=.  If it is ='in= instead of
   =nil=, hypothesis 3 is confirmed.

3. *Try =thinking_budget: 1024=* in request-params alongside
   =enable_thinking: t= to cap reasoning tokens and force a text
   response.

4. *Handle empty text gracefully*: if =final-text=""= and =:saw-reasoning=
   is set, either (a) expose reasoning text as the response, or (b)
   retry with a short follow-up prompt "Please summarise your findings
   in one sentence."
