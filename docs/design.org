* Introduction

Magent is an Emacs Lisp AI coding agent with a multi-agent architecture and permission-based tool access. It delegates all LLM communication to [[https://github.com/karthink/gptel][gptel]] (the sole external dependency beyond Emacs 27.1+).

This document provides a comprehensive overview of the magent architecture, design decisions, data structures, and implementation details.

For product positioning, especially the =live-in-emacs= thesis and the
=elisp-first= versus =bash-first= tradeoff, see
[[file:live-in-emacs-positioning.org][Live-in-Emacs Positioning]].

* Key Features

- Multi-agent system with specialized agents (build, plan, explore, general, compaction, title, summary)
- Permission-based tool access control with fine-grained rules per agent
- Custom agent support via YAML frontmatter + markdown files
- LLM integration via gptel (Anthropic Claude, OpenAI GPT, Ollama, Gemini, etc.)
- File operations: read, write, edit, grep, glob
- Shell command execution via bash tool
- Emacs Lisp evaluation via emacs_eval tool
- Agent delegation with explore and general subagents
- Web search via DuckDuckGo
- Claude Code skills for interacting with the running Emacs instance
- Session management with conversation history and buffer content persistence
- Single-request lock for serializing concurrent requests
- Minibuffer interface for quick prompts
- Transient menu for command discovery

* Core Modules

** Entry Point (magent.el)

The entry point defines ~magent-mode~, a minor mode with ~C-c m~ keybinding prefix. When enabled, it adds a modeline lighter showing spinner and busy state. Agent/skill initialization is deferred until first command use (~magent--ensure-initialized~).

Key components:
- ~magent-mode~: Minor mode with keybindings for all magent commands
- ~global-magent-mode~: Global minor mode for automatic activation
- ~magent--ensure-initialized~: Lazy initialization of agents and skills
- ~magent--spinner~: Modeline spinner for processing status
- ~magent--lighter~: Modeline construct showing current agent, busy state, and spinner

Package requirements: ~emacs 27.1+~, ~gptel 0.9.8~, ~spinner 1.7.4~, ~transient 0.4~.

** Configuration (magent-config.el)

All magent-specific configuration is in the ~magent~ customize group. LLM provider, model, and API key settings are managed entirely by gptel.

33 defcustom variables organized by category:

Core settings:
- ~magent-system-prompt~: System prompt for the AI agent (includes org-mode formatting instructions)
- ~magent-buffer-name~: Output buffer name (default: ~*magent*~)
- ~magent-auto-scroll~: Auto-scroll on new content

- ~magent-project-root-function~: Custom project root function
- ~magent-max-history~: Max session messages (default: 100)
- ~magent-request-timeout~: LLM request timeout in seconds (default: 120)
- ~magent-enable-logging~: Enable ~*magent-log*~ buffer
- ~magent-default-agent~: Default agent name (default: "build")
- ~magent-load-custom-agents~: Load agents from ~.magent/agent/*.md~

UI tag settings:
- ~magent-assistant-prompt~: Tag for assistant sections (default: "ASSISTANT")
- ~magent-user-prompt~: Tag for user sections (default: "USER")
- ~magent-tool-call-prompt~: Tag for tool calls (default: "tool")
- ~magent-error-prompt~: Tag for error sections (default: "error")

Path settings:
- ~magent-agent-directory~: Relative path to custom agent directory
- ~magent-session-directory~: Session persistence directory

Tool settings:
- ~magent-grep-program~: Path to ripgrep (auto-detected)
- ~magent-grep-max-matches~: Max grep matches (default: 100)
- ~magent-bash-timeout~: Bash command timeout (default: 30s)
- ~magent-emacs-eval-timeout~: Emacs eval timeout (default: 10s)

UI display settings:
- ~magent-ui-header-strike-through~: Enable strike-through line after section headers (default: nil)

- ~magent-ui-result-max-length~: Max tool result display length (default: 200)

- ~magent-ui-batch-insert-delay~: Streaming batch delay (default: 0.05s)

- ~magent-include-reasoning~: How to handle reasoning blocks (t/ignore/nil)
- ~magent-fsm-backend~: FSM backend (gptel or magent)

Also provides shared utilities:
- ~magent-project-root~: Project root detection (tries ~magent-project-root-function~, then projectile, then project.el, then ~default-directory~)
- ~magent-log~: Logging stub (overridden by ~magent-ui.el~ with real implementation)

** Queue System (magent-queue.el)

Single-request lock for serializing concurrent requests. Global singleton because magent has a single output buffer and single processing slot.

Data structure:
- ~magent-queue-item~: Struct with prompt (string), source (symbol), timestamp (float)

Queue behavior:
- When idle: prompt dispatches immediately
- When processing: prompt is rejected with a busy message

Key functions:
- ~magent-queue-enqueue~: Immediate-dispatch or reject if busy
- ~magent-queue-dequeue-and-run~: Release the processing lock
- ~magent-queue-clear~: No-op kept for API compatibility

The queue owns the processing lock (~magent-queue--processing~). ~magent-ui-process~ delegates to ~magent-queue-enqueue~, and ~magent-ui--run-item~ handles actual dispatch.

** Session Management (magent-session.el)

Sessions store conversation history with per-agent assignment and buffer content persistence.

Data structure:
- ~magent-session~: Container for messages, assigned agent, history settings, and saved buffer text
- ~magent-msg~: Individual message accessed via ~magent-msg-role~ and ~magent-msg-content~ accessors (not a struct, an alist)

Key features:
- Automatic message trimming when exceeding ~magent-max-history~ (lazy trim, triggers at max + 10)
- Buffer content persistence via ~buffer-content~ field for lossless restore
- Conversion to gptel prompt list format for API calls (~magent-session-to-gptel-prompt-list~)
- Session summarization (~magent-session-summarize~)

The agent system consists of two files:

- Hash-table registry with lookup, filtering, and selection
- Feature aliases for ~magent-agent-info~ and ~magent-agent-types~

** magent-agent.el

The agent processing module:
- ~magent-agent-process~: Main entry point for agent requests
- Builds gptel prompt list from session history
- Resolves agent (explicit > session > registry default)
- Applies per-agent overrides (model, temperature)
- Appends instruction-type skill prompts to system prompt
- Filters tools by permission rules
- Creates FSM and starts execution

** Permission System (magent-permission.el)

Rule-based access control with support for:
- Tool-level permissions: allow, deny, ask
- File-pattern rules with glob syntax
- Session-level overrides from user prompts
- Permission merging for composing rulesets

Permission resolution order:
1. Session overrides (always allow / always deny — set by user responses to [A]lways/[D]eny prompts)
2. Tool-specific rule (exact match)
3. Nested file rules (glob patterns against basename and full path)
4. Wildcard fallback (~*~)
5. Default allow

Session overrides are cleared on session reset, stored in ~magent-permission--session-overrides~.

Default permissions (~magent-permission-defaults~):
- ~*~ (wildcard): allow
- ~bash~: ask
- ~emacs_eval~: ask
- ~delegate~: allow
- ~web_search~: allow
- ~read~: allow, except ~*.env~ and ~*.env.*~ (deny), ~*.env.example~ (allow)
- ~write~: ask, except ~*.env~ (deny), ~*.env.example~ (allow)
- ~edit~: allow, except ~*.env~ (deny), ~*.env.example~ (allow)

Example permission rules (using permission key symbols, not gptel tool names):
#+begin_src elisp
'((read . allow)
  (write . (("*.env" . deny)
            (* . ask)))
  (bash . ask))
#+end_src

** Tool System (magent-tools.el)

10 tools implemented as gptel-tool structs:

| Tool          | Async | Confirm | Description                              |
|---------------+-------+---------+------------------------------------------|
| read_file     | yes   | no      | Read file contents                       |
| write_file    | yes   | yes     | Write/create file (auto-creates dirs)    |
| edit_file     | yes   | yes     | Replace exact text in file               |
| grep          | yes   | no      | Regex search via ripgrep (async process) |
| glob          | yes   | no      | Find files matching glob pattern         |
| bash          | yes   | yes     | Execute shell commands (async process)   |
| emacs_eval    | yes   | yes     | Evaluate Emacs Lisp expressions          |
| delegate      | yes   | no      | Spawn nested request using subagent      |
| skill_invoke  | yes   | no      | Invoke Claude Code skills                |
| web_search    | yes   | no      | Search the web via DuckDuckGo            |

All tools are asynchronous (callback-based). Tools with ~:confirm t~ prompt user confirmation before execution.

Tools are globally defined but filtered per-agent via:
1. ~magent-enable-tools~ global config (which tools are available at all)
2. Agent permission rules (which tools this agent can use)
3. ~magent-permission-tool-available-p~ handles nested rules (e.g., plan agent where edit defaults to deny but specific paths are allow)

Tool name to permission key mapping:

  ("write_file"   . write)
  ("edit_file"    . edit)
  ("grep"         . grep)
  ("glob"         . glob)
  ("bash"         . bash)
  ("emacs_eval"   . emacs_eval)
  ("delegate"     . delegate)
  ("skill_invoke" . skill)
  ("web_search"   . web_search))
#+end_src

** Finite State Machine (magent-fsm.el + backends)

Unified entry point for the tool-calling loop with two switchable backends controlled by ~magent-fsm-backend~:

*** Unified API

- ~magent-fsm-create~: Create FSM with keyword args
- ~magent-fsm-start~: Start FSM execution
- ~magent-fsm-abort~: Abort FSM execution
- ~magent-fsm-destroy~: Clean up resources

All dispatch to the configured backend.

*** Native Backend (magent-fsm-backend-native.el)

Custom FSM with full state machine control and permission-aware confirmation.

States:
- ~INIT~: Validate inputs (session, backend, model)
- ~SEND~: Prepare and fire HTTP request via gptel, start timeout timer
- ~WAIT~: HTTP request in flight; gptel handles tool-calling loop
- ~PROCESS~: Parse response, check for tool usage
- ~TOOL~: Execute pending tool calls with mutex-based synchronization
- ~DONE~: Terminal state (success) — records response in session, calls callback, destroys FSM
- ~ERROR~: Terminal state (failure) — calls callback with nil, destroys FSM

State transitions:
#+begin_example
INIT -> SEND -> WAIT -> PROCESS -> TOOL -> SEND -> WAIT -> ... -> DONE
                                    |                     ^
                                    +---------------------+
#+end_example

FSM callback handles:
- Reasoning blocks (streaming ~(reasoning . text)~ cons cells)
- Streaming text chunks
- Tool call confirmations (~(tool-call . ...)~ cons cells)
- Tool result injection (~(tool-result . ...)~ cons cells)
- Multiple rounds of tool calls
- Timeout handling via timer
- Buffer-boundary error suppression (evil-mode cursor adjustments)

Tool confirmation flow:
1. Check session override (always allow / always deny)
2. Check file-specific deny rules
3. Check file-specific allow rules
4. Prompt user interactively: [y]es / [n]o / [A]lways allow / [D]eny always

Tool function wrapping: ~magent-fsm--wrap-tool-function~ wraps each tool to display call/result in the UI before/after execution.

*** Gptel Backend (magent-fsm-backend-gptel.el)

Lightweight wrapper around ~gptel-request~ that delegates the entire tool-calling loop to gptel.

- Uses a hidden request buffer
- Tools auto-execute without interactive confirmation (gptel's confirm UI requires a visible buffer)
- No custom state machine — just sets up the request and handles the callback
- Streaming mode enabled by default
- Handles tool-use rounds by calling ~magent-ui-continue-streaming~ and final rounds by calling ~magent-ui-finish-streaming-fontify~

** Skills System (magent-skills.el + skill files)

Two skill types:
- ~instruction~ type: Markdown body injected into system prompt via ~magent-skills-get-instruction-prompts~
- ~tool~ type: Invoked via ~skill_invoke~ tool with predefined operations

Built-in ~emacs~ skill provides 6 operations:
- ~list-functions~: List interactive commands matching prefix
- ~describe-function~: Get function documentation
- ~eval-expression~: Evaluate arbitrary elisp
- ~execute-keys~: Simulate keystrokes (kbd format)
- ~minibuffer-prompt~: Read current minibuffer state
- ~current-buffer-state~: Get buffer name, mode, and content

Skills loaded from:
- Global: ~~/.emacs.d/magent-skills/<name>/SKILL.md~
- Project: ~.magent/skills/<name>/SKILL.md~

Tool-type skills can have companion ~.el~ files defining ~magent-skill-<name>-invoke~ function. The companion file is discovered automatically (~SKILL.el~ or ~<skill-name>.el~ in the same directory).

** Markdown-to-Org Conversion (magent-md2org.el)

Converts markdown syntax from LLM output into org-mode syntax for rendering in the output buffer. Called during streaming to convert assistant responses in-place.

Conversions:
- Headers: shifted +1 level (~#~ → ~**~, ~##~ → ~***~, etc., capped at 6 levels)
- Fenced code blocks: ~```lang~ → ~#+begin_src lang~ / ~#+end_src~ (handles indented fences)
- Bold: ~**text**~ → ~*text*~ (uses sentinel chars to avoid collision with italic)
- Italic: ~*text*~ → ~/text/~
- Inline code: ~`code`~ → ~~code~~

Preserves ~#+begin_tool~ / ~#+end_tool~ and ~#+begin_think~ / ~#+end_think~ blocks unchanged (skips their contents entirely).

Key functions:
- ~magent-md2org-convert-region~: In-place conversion of a buffer region
- ~magent-md2org-convert-string~: Convenience wrapper returning converted string

** UI Layer (magent-ui.el)

Output buffer derives from ~org-mode~ (~magent-output-mode~) for native folding support:
- Level-1 headings (~*~) for message section markers (user, assistant, tool, error)
- Level-2+ headings (~**~) for LLM content
- ~#+begin_tool~ / ~#+end_tool~ blocks for tool calls (auto-folded on completion)
- ~#+begin_think~ / ~#+end_think~ blocks for reasoning (auto-folded)
- Section folding with ~TAB~ / ~S-TAB~ via org-cycle

Streaming support:
- Batched chunk insertion to reduce UI updates (~magent-ui-batch-insert-delay~)
- Streaming state tracking: section start, has-text flag, batch buffer, batch timer
- Empty streaming sections (tool-only rounds) are cleaned up
- ~magent-ui-continue-streaming~ for multi-round tool use within one section
- ~magent-ui-finish-streaming-fontify~ finalizes and removes orphaned headings

Buffer management:
- ~magent-ui-render-history~: Restores buffer from saved content or re-renders from session messages
- Buffer content saved to session's ~buffer-content~ field for lossless restore
- ~magent-ui--revert-buffer~: Revert function delegates to render-history

Transient menu (~magent-transient-menu~, bound to ~?~ in output buffer):
- Session: prompt, clear session
- Agent: select, show current, list
- Queue: view queue, remove item
- Buffer: toggle read-only
- Control: interrupt (C-g)
- Logs: show log, clear log

* Key Design Decisions

** No Custom HTTP Client

All LLM communication goes through gptel. This decision:
- Reduces code maintenance burden
- Ensures compatibility with all gptel providers
- Leverages gptel's battle-tested HTTP implementation
- Simplifies tool-calling loop implementation

** Switchable FSM Backends

~magent-fsm-backend~ (default ~gptel~) selects between:
- *Native backend*: Full FSM control with custom state machine. Provides permission-aware interactive confirmation prompts. More complex but offers fine-grained control over tool execution flow.
- *Gptel backend*: Lightweight wrapper that delegates the entire tool-calling loop to gptel. Simpler code path but tools auto-execute without interactive confirmation (gptel's confirm UI requires a visible buffer, and this backend uses a hidden request buffer).

** Per-Agent gptel Overrides

The ~magent-agent-info-apply-gptel-overrides~ function temporarily sets gptel variables for the duration of a request. The agent's MODEL field can be:
- nil: use global ~gptel-backend~ and ~gptel-model~
- A gptel backend object: use it with global ~gptel-model~
- A cons (BACKEND . MODEL-SYMBOL): use both overrides

Temperature is overridden if the agent specifies it.

** Agent Modes

The ~primary~ vs ~subagent~ distinction enables:
- Clear API between agent types
- Permission control (non-subagents cannot be delegated to via ~delegate~ tool)
- UI filtering (only primary agents shown in agent selection)

** Tool Filtering

Tools are defined once globally but filtered per-agent via permission rules. This approach:
- Avoids duplicate tool definitions
- Centralizes tool implementation
- Enables dynamic permission changes per session

** Skill Types

Two distinct skill types serve different use cases:
- ~instruction~ type: Claude Code style — prompts injected into system message. LLM follows instructions and uses available tools directly.
- ~tool~ type: Traditional — invoked via ~skill_invoke~ tool with predefined operations. Companion ~.el~ files provide the implementation.

** Session Buffer Content Persistence

Sessions save the raw buffer content (not just messages) for lossless restore. This preserves:
- Tool call blocks and their results
- Reasoning blocks
- Error messages
- Formatting that cannot be reconstructed from messages alone

When the buffer is reverted or re-rendered, saved content is restored directly rather than re-rendering from session messages.

** Prompt Queue

The queue serializes concurrent requests because magent has a single output buffer and single processing slot. Design choices:
- Global singleton (not per-session)
- FIFO ordering
- Immediate dispatch when idle, queuing when busy
- Configurable max size with error on overflow
- Auto-dispatch next item after completion or interruption
- Deferred dispatch via ~run-at-time 0~ to avoid stack depth issues

** Streaming Support

Streaming is handled at both the HTTP layer (gptel) and UI layer (magent-ui):
- gptel manages HTTP streaming and tool-calling within streaming
- magent-ui batches small chunks to reduce UI updates
- Empty streaming sections (tool-only rounds) are cleaned up
- Multi-round tool use stays within the same org section (~magent-ui-continue-streaming~)

* Integration with gptel

Magent delegates all LLM communication to gptel:

#+begin_src elisp
(gptel-request
    prompt-list
  :buffer request-buffer
  :system system-prompt
  :stream streaming
  :callback fsm-callback)
#+end_src

gptel handles:
- HTTP request/response
- Provider-specific APIs (Anthropic, OpenAI, Gemini, Ollama, etc.)
- Tool calling loop (invoking tools, injecting results, continuing)
- Streaming
- Error handling

Magent adds:
- Multi-agent architecture with specialized agents
- Permission-based tool access with file-pattern rules
- Session management with buffer content persistence
- Org-mode based UI rendering with streaming
- Single-request processing lock
- Skill system (instruction and tool types)
- Interactive tool confirmation with session overrides
